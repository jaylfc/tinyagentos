# TAOS v0.2 release notes (draft)

Headline: **TurboQuant KV cache compression is in, and the cluster
protocol now speaks asymmetric K/V natively.**

This release lands the work from the overnight TurboQuant push
(2026-04-11) plus a clutch of scheduler, worker and UX improvements.

## Headline features

### TurboQuant end to end

- Worker agent detects and advertises `kv_cache_quant_k_support` and
  `kv_cache_quant_v_support` per backend, plus a boundary-layer flag.
- Cluster protocol unions those into a detailed options response on
  `GET /api/cluster/kv-quant-options`.
- Deploy wizard renders K, V and boundary-layer controls only when at
  least one online worker advertises support — no dead controls on
  fp16-only clusters.
- llama-cpp workers use the `TheTom/llama-cpp-turboquant` fork
  automatically when built from source. Benchmark numbers on a single
  RTX 3060 reached **786K context** on Qwen3.5-9B-Q4_K_M using T2/T2
  (up from 131K baseline) with decode speed stable at 56-62 t/s.
- Per-model KV quant quality gate (`scripts/kv_quant_validator.py`)
  ships as the needle-in-haystack validator, with docs explaining when
  to run it and how to interpret the results.
- Asymmetric K/V is the default: Q8 keys + T3 values is the
  recommended safe config on every Qwen3.5 model we tested.

### Agent failure handling

- Per-agent `on_worker_failure` policy with three options: `pause`,
  `fallback` (try next model in list), `escalate-immediately`.
- `fallback_models` list per agent, persisted in config.
- Deploy wizard exposes a failure policy step with sensible defaults.
- Paused agents show a badge and a Resume button in the desktop UI.
- Agent can swap its underlying model at runtime via `/agent/<id>/model`.

### Core-aware scheduler

- New `CoreAwareModelScheduler` tracks NPU core allocation on RK3588
  and similar multi-core accelerators.
- Shrink-reload and evict-reload policies let the scheduler squeeze an
  extra model in without killing an in-flight request.
- Resource-shape module describes each backend's per-model footprint
  so the scheduler can reason about placement without guessing.

### Cluster + install improvements

- Worker re-registration on controller restart (#166). Heartbeats that
  come back 404 now trigger an automatic re-register instead of
  silently dropping the worker from the cluster.
- `install-worker.sh` bundles a TAOS-namespaced Ollama on port 21434
  so it coexists with any pre-existing user Ollama on 11434.
- Mirror policy: Rockchip install artifacts served from a
  TAOS-controlled HuggingFace repo so `install-rknpu.sh` does not
  break when upstream drops a build.
- Model resolver prefix match tightened: `qwen3` no longer matches
  `qwen3.5` (#221).
- Framework registry with verification statuses — untested frameworks
  render greyed out in the wizard instead of lying about support.

### UX

- Models app aggregates controller, worker and cloud models into one
  view with variant/quant filtering.
- Cluster widget shows potential capabilities derived from hardware
  tier, so you can see what a worker *could* do once backends are
  installed — not just what it currently advertises.
- Activity app gets a Cluster panel; new standalone Cluster management
  app for the workers detail view.
- Models app + deploy wizard route deploy requests to whichever worker
  already holds the requested model.

## Under the hood

- Split `kv_cache_quant` into `kv_cache_quant_k` and
  `kv_cache_quant_v` plus `kv_cache_quant_boundary_layers` across
  config, cluster protocol, worker probe and UI. Legacy configs
  migrate transparently — the old single field is copied into both K
  and V on load.
- `WorkerInfo` carries the new fields; the worker protocol validates
  them with pydantic defaults so legacy workers still register.
- Scheduler gets a `get_resource_shape` hook on `BackendEntry` for the
  core-aware path.
- Em-dash scrubbing pass across scheduler and kv-quant plumbing files
  (prefer ASCII in source).
- Test suite: 1200+ tests, all passing. `TestHeartbeat` updated to
  assert on int status code rather than bool after #166.

## Platform + cloud prep

- LXC provisioning scripts for tinyagentos.com (#90)
- Caddyfile for the platform LXC (#91)

## Known issues

- Rockchip NPU does not yet support TurboQuant — requires rkllm SDK
  feature request, tracked in #193
- vLLM TurboQuant fork (`0xSero/turboquant`) is a hybrid-mode no-op on
  0.19.0; llama-cpp is the only production TurboQuant path today (#224)
- Network model placement between workers still stubbed; lands in
  Phase 1.5 (#176)

## Upgrade notes

- Workers built against the old single-field `kv_cache_quant` will
  keep working; the controller migrates their advertised values on
  first heartbeat.
- Config files with the legacy field get normalised on load. A save +
  reload round-trip writes the new split fields and drops the old
  key.
- No breaking API changes. The `/api/cluster/kv-quant-options`
  response still carries the legacy flat `options` list alongside the
  new `k`, `v`, and `boundary_layer_protect` fields, so any pre-split
  desktop builds continue to function.
