# Worker Pairing & Sharing — Design Spec

## Overview

Two related capabilities for the taOS cluster:

1. **Pairing** — link a remote worker to a taOS install securely. Today this works for self-hosted users via direct manual config; this spec adds a proper OTP flow that works without editing config files. Cloud-account users get the same pairing plus auto-discovery via Headscale.
2. **Sharing** — the owner of a paired worker can grant other users (friends, family, teammates, studio collaborators) access to that worker. Sharing is per-recipient, with trust tiers, resource limits, and clean revoke.

This spec is for **taOS** (the platform). It's not a taosmd feature.

Sharing is the load-bearing new capability. Pairing is a prerequisite.

## Why this exists

People have spare hardware. A gaming PC that sits idle 16 hours a day. A studio with a render workstation that's quiet on weekends. A friend with a Mac Studio who'd happily lend you 8 hours of MLX time. Today there's no clean way for any of those people to lend that compute to a friend who's running on a Pi.

The choices today are: give them SSH and trust they don't break anything, or rebuild your friend's stack on your hardware. Both are awful. Sharing is the obvious feature that turns a single-user platform into something you can pool with the people you trust, without becoming a tiny cloud provider.

The pairing flow makes the prerequisite painless. A 10-minute OTP from the worker, a paste into taOS, done — the same UX whether you're on the LAN or across the world (Headscale handles the network).

## Glossary

- **Worker** — a machine running the taOS worker agent, exposing inference/training compute to one or more controllers.
- **Owner** — the user who paired a worker to their taOS account. Workers have exactly one owner.
- **Recipient** — a user (or specific agent) who's been granted use of someone else's worker via a share.
- **Share** — a persistent grant from owner → recipient with a trust tier, capability set, and limits.
- **Pairing** — the act of linking a worker to a taOS install for the first time.
- **OTP** — short-lived one-time password generated by the worker, used for pairing.
- **Share token** — short-lived cryptographic credential used to bootstrap a new share. Distinct from an OTP.
- **Bearer credential** — long-lived credential issued at share creation, held by the recipient's instance, used per-call to authenticate.

## Pairing (foundation)

### Self-hosted OTP pairing

Worker generates an 8-digit OTP via CLI or tray icon. OTP is valid for 10 minutes, single use, scoped to one specific controller URL the user paired against (so a leaked OTP can't be used to add the worker to a different controller).

Generation:

```bash
# CLI
tinyagentos-worker pair --controller https://taos.local:6969
# → OTP: 12345678 (valid 10 minutes, paired against https://taos.local:6969)
```

Tray icon equivalent: "Pair this worker..." menu item opens a dialog with a controller URL field and a "Generate code" button.

Owner pastes the OTP into their taOS Cluster app. taOS calls the worker's `/pair` endpoint with the OTP. Worker validates, returns a long-lived bearer token, persists the (controller, token) pair. Subsequent worker calls authenticate with that bearer token; OTP is invalidated immediately on first use.

### Cloud-account pairing

For users with a `tinyagentos.com` cloud account: the worker can register itself against the user's account directly, without an OTP, by signing in once. This piggybacks on Headscale for NAT traversal so the controller doesn't need a public IP. The OTP path still works on the same worker; cloud-account is just the easier UX.

Both paths produce the same kind of pairing record on the controller side. The difference is purely how the initial bearer credential was bootstrapped.

## Sharing

### Trust tiers

The owner picks a tier per share. Tiers are tier-A by default; B and C are explicit upgrades.

**Tier A — Inference only**
- Recipient's agent calls run against models loaded on the owner's worker (chat, embed, rerank, image-gen if allowed).
- No code runs on the owner's hardware that the recipient supplied.
- Smallest blast radius. Cryptographic enforcement at the worker.

**Tier B — Inference + scoped writes**
- Above, plus the recipient can install models from the catalog into a quota-limited shared workspace, fine-tune LoRAs on data they upload, write checkpoint files to their share-scoped directory.
- No arbitrary code execution; no host filesystem access outside the share-scoped directory.

**Tier C — Full agent host**
- Above, plus the recipient can deploy full LXC/Docker containers to the worker — run MCP servers, execute shell commands inside their container.
- Requires per-share container isolation. v1 ships LXC; Firecracker is a follow-up for higher-trust scenarios.

The default for any new share is A. Upgrade to B or C is per-share, explicit, and warned about ("This recipient will be able to write data to your worker. Continue?").

### Identity model

Hybrid — both work, cloud account is the easy path:

**Cloud-account users**: discover each other via username on `tinyagentos.com`. Friends list / contacts model (no friend-request graph in v1; both parties are simply discoverable by default with an opt-out toggle). Sharing routes through Headscale for NAT traversal.

**Self-hosted P2P**: same share record, different transport. Owner generates a share invite (token + worker public key), hands it to recipient out-of-band (Signal, email, in person). Recipient pastes into their taOS. Their instance dials the worker directly using the embedded public key for TLS pinning.

The cloud account is **strictly optional** at every layer. The implementation is "two transports for the same share record" — never "the cloud account is required for X". This protects the self-hosted-first promise.

### Recipient granularity

A share record names either:

- **A user** — that user's agents (any of them) can use the worker through their taOS instance.
- **A specific agent** — only that named agent on the recipient's side can use the worker.

The user case covers most scenarios (friends/family/teammates). The specific-agent case lets the owner say "yes I'll lend my GPU but only to your research bot, not your fifty experimental agents".

Recipient instances can additionally enforce per-agent gating on their side via the librarian config — but that's their choice, not the owner's enforcement.

### Limits

The full v1 surface, expressible per share:

| Limit | v1 default | Notes |
|---|---|---|
| Trust tier | A | A / B / C |
| Max concurrent jobs | 1 | Stops a single recipient from saturating the worker |
| Capability allowlist | `chat, embed` | Opt-in for `image-gen`, `training`, `rerank`, `multimodal` |
| Storage quota (GB) | 0 (tier A); 5 (tier B); 20 (tier C) | Per-share cap on disk usage |
| Encrypted volumes | optional, recipient opts in | Recipient brings the key; owner can't read |
| Expiry | `inactive_30d` auto-revoke | Configurable; 30-day-inactive default kills zombie shares |
| Pause without revoke | enabled | Disable share temporarily, keep config |

**Worker-level safety net** (independent of any one share):

- **Free-disk floor**: owner sets minimum free disk the worker reserves for its own work. Default: 20% of disk OR 50 GB, whichever is larger. Any share operation that would breach the floor is refused with a clear error naming the responsible share.
- **Optional**: "Auto-evict cached models LRU-style when free space would breach the floor" — off by default.

**Follow-ups (filed as issues, not v1)**:

- Max VRAM/RAM per share
- Time windows ("only between 23:00–07:00 local")
- Compute budget per period
- Model allowlist (more granular than capability allowlist)
- Network policy (only meaningful at tier C)
- Ranked priority between sharers

**Out of scope forever (public)**:

- Per-recipient billing (monetisation surface; lives on the private side)
- Geographic restrictions

### Share creation

Both directions ship in v1.

**Owner-issued (default UX, A from brainstorm)**:

1. Owner opens worker in Cluster app, clicks "Share with...".
2. Configures recipient (user or agent), tier, limits, expiry.
3. taOS generates a single-use share token (signed claim about `(worker_id, recipient_identity, capabilities, expires_at, tier)`, 24h TTL).
4. Owner hands the token to recipient out-of-band (or auto-delivered if both are cloud-account users).
5. Recipient pastes into their taOS. Their instance calls owner's worker `/share/redeem` with the token.
6. Worker validates, creates the persistent share record, returns a long-lived bearer credential.
7. Recipient's instance persists the credential and the worker now appears in their cluster view (with a "borrowed" badge).

**Recipient-requested (also v1)**:

1. Recipient generates a "request to use compute" token in their taOS — names the capability they need (`image-gen`, `chat`, etc.).
2. Sends to owner.
3. Owner sees a "Jay wants compute for image-gen" notification in their taOS, picks which worker to grant, configures tier + limits + expiry, clicks Approve.
4. Credential is delivered back automatically over the share connection.

Both flows produce the same kind of share record. Different bootstrapping; identical lifecycle thereafter.

### Lifecycle: revoke and expiry

**In-flight jobs at revoke** — owner picks per-revoke at the moment, with sensible defaults:

- **Drain (default)**: new requests refused, in-flight requests run to completion with a hard ceiling (60s for inference, 5min for training).
- **Hard kill (option)**: in-flight requests die immediately, recipient gets 503. Used for "I'm gaming and I need the GPU NOW" scenarios.

**Persistent data on revoke** — owner picks per-revoke:

- **Soft-delete with 7-day grace (default)**: data marked deleted, recoverable if owner re-shares within 7 days.
- **Hard-delete now (option)**.

**Cached models on revoke**:

- **Reference-counted always**. Models the recipient triggered downloads of are kept on the worker, eligible for normal LRU eviction once no active share references them. Friendly to multi-recipient workers that share popular models.

**Encrypted volumes on revoke**:

- **Volume goes opaque immediately** (the key was the recipient's; without it the volume is just ciphertext on the host). Underlying file lifecycle still follows the persistent-data rule (soft-delete or hard-delete) but the data inside is inert either way.

**Auto-expiry triggers**:

- Inactive for the share's expiry window (default 30 days)
- Storage quota breached and recipient won't free space within 24h notice
- Compute budget exhausted (when budgets ship)

Both sides get a notification 24h before expiry hits so it's never a surprise.

### Container isolation (tier B/C)

**v1: LXC** — matches existing taOS infrastructure; the scheduler already knows about it. Trust assumption is "people you actually know" — friends, family, colleagues, contractors. LXC stops accidents but not a determined attacker.

**Follow-up: Firecracker microVMs** as opt-in per-share or per-worker setting, for "share with strangers / accept paid workloads" scenarios. Firecracker on ARM (Pi/Apple Silicon) is rough; gVisor on GPU workloads adds significant overhead. Defer until there's real demand.

The spec is honest about this: README will say "today, share at tier B/C only with people you trust; for arms-length sharing, wait for the Firecracker work in #X".

### Encrypted volumes

**Promise**: owner provides storage, can't read the data. Recipient brings the key.

**Threat model — protects against**:

- ✅ Passive snooping by the owner (`cat` on the volume file shows ciphertext).
- ✅ Owner sharing your data with a third party (they hand over ciphertext).

**Does NOT protect against**:

- ⚠️ Active root-level adversary on the worker (could dump RAM while volume is mounted and steal the key).
- ⚠️ Active host kernel modification.

Removing those last two needs confidential computing (TEE / SEV-SNP / TDX), not viable for v1 on consumer hardware. The honest pitch is **"protects against passive snooping and inadvertent leakage, not against a determined hostile root"** — adequate for friends/family/colleagues; not adequate for stranger-sharing.

### Implementation: LUKS-inside-container

Cross-platform compatible because tier B/C recipients always run inside a Linux container (LXC on Linux hosts, Linux VM via Docker Desktop / Lima / WSL2 on macOS / Windows hosts). The host OS just stores opaque files; the container handles crypto.

| Worker OS | Container runtime | What the host stores | Where LUKS runs |
|---|---|---|---|
| Linux | LXC or Docker | sparse file on host FS | inside the recipient's container |
| macOS | Docker Desktop / Lima | sparse file on macOS FS, passed through to the Linux VM | inside the recipient's container |
| Windows | Docker Desktop / WSL2 | sparse file on NTFS, passed through to the Linux VM | inside the recipient's container |

**Mount flow**:

1. Worker allocates a sparse file `/var/lib/taos/shares/{recipient}/{share_id}.img` of the share's storage quota size.
2. Worker passes the file as a block device to the recipient's container (`--device` for Docker, loop config for LXC, virtio-blk for VM-based runtimes).
3. Recipient's container starts. Recipient's taOS pushes the key over the existing share connection at mount time.
4. Container's mount helper runs `cryptsetup luksOpen` + `mount` inside its own namespace.
5. Recipient's processes see `/data` as plaintext; outside the container, it's an opaque file.

**Key lifecycle**:

- Recipient generates a 256-bit key on share creation.
- Stored encrypted-at-rest in recipient's own taOS (existing SecretsStore).
- Sent over the existing TLS share connection at mount time. **Never persisted on the worker** — held only in the mount helper's process memory.
- Scrubbed from worker memory on unmount, share revoke, share expiry, container shutdown, OR a heartbeat-loss timeout.
- **Key loss = data loss**. Documented loud in the recipient's UI when they create the share.

**Fallback for restricted environments**: when the recipient's container can't get a privileged mount (e.g. unprivileged LXC on a hardened Pi), fall back to **gocryptfs in userspace** inside the container. ~10% slower for big sequential reads but works without privileges. Same key flow.

**Edge cases**:

- macOS without Docker Desktop / Lima: tier B/C unavailable on this worker. Tier A still works. Worker marked "tier A only" in the share UI.
- Windows without Docker Desktop / WSL2: same — tier A only.

### Recipient UX

Shared workers appear in the recipient's normal **Cluster app worker list** alongside their own workers, with:

- A **"borrowed" badge** clearly marking it as someone else's hardware.
- A **coloured border** matching the share's identifying colour (auto-assigned per-owner so workers from the same owner all share a colour).
- **Owner attribution** ("borrowed from Jay since 3 Apr") on the worker card.
- **Recipient's own actions limited to what the share permits** — UI greys out tier B/C deploy actions on a tier A share, etc.

This keeps the borrowed worker discoverable and usable in the same flows as owned workers, but never ambiguous about who owns it.

### Owner observability

State changes surface as **notifications** (uses the existing notification stack):

- Share started (recipient's first call after creation)
- Approaching quota (storage 80%, compute 80%)
- Expiring soon (24h before)
- Expired / revoked (with cause)
- Heartbeat lost (recipient's instance unreachable for >5min)

Always-visible status sits in the **Activity app** as a "Shares" card alongside the Cluster card:

- One row per active share
- Recipient name + which worker
- Hours used this period / total
- Last call timestamp
- Quick "Pause" / "Revoke" buttons
- Click-through to a per-share usage history page

### Discovery for cloud-account users

Simple username-based lookup on `tinyagentos.com`:

- Each cloud account has a unique handle (set at signup).
- Discovery defaults to ON (you can search and be searched). Opt-out toggle in account settings.
- "Share with..." opens a contact picker that searches the cloud's handle directory.
- For people not on the platform yet, "Send invite link" generates a sign-up URL prefilled with the share token — they create an account and the share lands in their cluster view.

No friend-requests graph in v1. No privacy levels. No mutual-acceptance dance. Either you're discoverable or you're not.

## API surface

### Pairing

```
POST   /api/cluster/pair                  body: {otp, controller_url}
                                          → returns: {bearer, worker_id}
GET    /api/cluster/pair/status           polled by worker tray during pairing
DELETE /api/cluster/workers/{id}/pair     unpair a worker (owner action)
```

### Sharing — owner side

```
POST   /api/cluster/workers/{id}/shares                create a share (owner-issued)
                                                       body: {recipient, tier, limits, expiry}
                                                       → returns: {share_token}
GET    /api/cluster/workers/{id}/shares                list active shares for this worker
GET    /api/cluster/shares                             list all shares the owner has issued
PUT    /api/cluster/shares/{id}                        patch limits / pause / resume
DELETE /api/cluster/shares/{id}                        revoke (with body {drain: bool, hard_delete: bool})
GET    /api/cluster/shares/{id}/usage                  usage history + active jobs
```

### Sharing — recipient side

```
POST   /api/cluster/shares/redeem                      paste a share token to redeem
                                                       body: {share_token}
                                                       → returns: {worker_metadata, bearer}
GET    /api/cluster/borrowed                           list workers shared TO this user
DELETE /api/cluster/borrowed/{id}                      stop using a borrowed worker (recipient declines)
POST   /api/cluster/borrowed/{id}/key                  push encrypted-volume key for mount
```

### Sharing — request flow (recipient-initiated)

```
POST   /api/cluster/share-requests                     recipient generates a request token
                                                       body: {capability, justification}
                                                       → returns: {request_token}
POST   /api/cluster/share-requests/redeem              owner pastes/accepts a request
                                                       body: {request_token, worker_id, tier, limits, expiry}
GET    /api/cluster/share-requests                     list pending requests (both directions)
```

## Frontend

### Cluster app additions

- **Worker card**: "Share with..." button (owner only). Borrowed badge + coloured border for shared-to-me workers.
- **Share dialog**: recipient picker (cloud handle search OR self-hosted invite mode), tier selector, capability checklist, storage quota slider, expiry picker, "Generate token" button.
- **Worker settings**: free-disk floor slider, "auto-evict on breach" toggle, list of active shares with quick-pause/revoke.

### Activity app additions

- **Shares card** alongside the Cluster card. One row per active share. Quick-pause and revoke buttons.
- **Per-share usage page**: hours used, requests served, daily breakdown chart, currently-running jobs.

### Cluster recipient view

- Shared workers in the same list as owned workers. Badge + border + owner attribution.
- Greyed-out actions for capabilities the share doesn't permit.
- "Stop using this worker" option (recipient-side decline; doesn't affect owner's record).

### Power menu

- Existing: Lock / Settings / Restart / Sign out
- New: "Generate pairing code" (when on a tray-icon worker app) — opens the pairing OTP modal.

## Security model summary

| Layer | What's enforced | How |
|---|---|---|
| Pairing | Only the owning controller can use a worker | OTP → bearer token, scoped to controller URL |
| Share token | Single-use, 24h TTL | Cryptographic signature; worker rejects expired/replayed tokens |
| Bearer credential | Long-lived, revocable | Worker maintains a revocation set; one DELETE invalidates immediately |
| Tier A enforcement | No recipient code on owner hardware | Worker validates each call's capability against share's tier+allowlist |
| Tier B/C enforcement | Recipient code runs in container | LXC / Docker container with cgroup limits; Firecracker as follow-up |
| Storage quota | Per-share cap | Worker checks before any write; refuses with named-share error |
| Worker disk floor | Owner-reserved free space | Pre-write check at the worker layer |
| Encrypted volumes | Owner can't read recipient data | LUKS / gocryptfs inside container, key never persisted on worker |
| Revoke | Immediate effect | Bearer added to revocation set; in-flight handled per drain/kill choice |

## Out of scope (v1)

- Firecracker microVM isolation (follow-up issue)
- Recipient-initiated share requests with marketplace dynamics (requests work, but there's no "looking for compute, anyone?" board)
- Per-period compute budgets (need metering infra first)
- Time windows (need scheduling layer)
- Geographic restrictions (not what this product is for)
- Per-recipient billing (monetisation; private notes only)
- Confidential computing for stranger-sharing (TEE / SEV-SNP / TDX)
- Cross-controller share inheritance (if owner's controller dies, the share is dead — no automatic failover)

## Implementation order

Phased so each phase ships something useful on its own.

### Phase 1: Pairing + tier A sharing

- OTP generation in worker tray + CLI
- `/api/cluster/pair` flow
- Cluster app pairing UI
- Trust tier A only
- Owner-issued share creation
- Capability allowlist + concurrent-jobs limit
- Per-share expiry (`inactive_30d` default) + manual revoke
- Recipient borrowed-worker UI (badge + colour border)
- Owner observability (notifications + Activity card)
- Discovery for cloud-account users

This is the smallest end-to-end useful slice. Friends-and-family GPU lending works at this point.

### Phase 2: Tier B + storage + encryption

- Trust tier B (scoped writes)
- Storage quotas + worker disk floor
- Encrypted volumes (LUKS-inside-container; gocryptfs fallback)
- Pause without revoke

### Phase 3: Tier C + container isolation

- Trust tier C (full agent host)
- Tier C container management on the worker side
- Audit logging surface

### Phase 4: Recipient-requested shares + Headscale relay

- Recipient-initiated request flow
- Cloud-account contact picker
- Headscale-routed connections for cloud-account users behind NAT

### Phase 5 (follow-ups, scoped at the time)

- Firecracker microVM isolation
- Time windows, compute budgets, model allowlist
- Ranked priority between sharers
- Cross-controller share inheritance
- Confidential computing path
